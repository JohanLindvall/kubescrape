## Commands

```sh

make test               # go test ./...
go test ./internal/store/ -run TestWaitIsPerContainer -v   # single test
make vet fmt tidy
make lint               # golangci-lint v2 (auto-installed to GOPATH/bin; config .golangci.yml)
```

### End-to-end verification (kind)

Changes to informer wiring, the store, or the HTTP API should be verified against a real cluster, not just unit tests:

```sh
make cluster-up         # 3-node kind cluster; downloads kind/kubectl into hack/bin; deploys hack/test-workloads.yaml
go run ./cmd/kubescrape # uses the kind kubeconfig context
make image && hack/bin/kind load docker-image ghcr.io/johanlindvall/kubescrape:latest --name kubescrape
kubectl apply -f deploy/kubernetes.yaml   # in-cluster deployment (namespace: monitoring)
make cluster-down
```

`hack/test-workloads.yaml` intentionally exercises every code path: a pod-annotated Deployment (ReplicaSet→Deployment chain, pod-source targets), a non-annotated Deployment behind an annotated Service with a named `targetPort` (service-source targets), a CronJob (Job→CronJob chain), plus a StatefulSet and a DaemonSet (direct-owner chains).

## Public packages (`pkg/`)

`pkg/promparse` (the exposition parser — `internal/agent/promscrape` aliases its types in `promparse.go`), `pkg/spool`, `pkg/kubemeta`, `pkg/metaclient`, `pkg/logattrs`, `pkg/otlpsplit` (size-bounded OTLP payload splitting) and `pkg/cgroupid` (cgroup path → pod/container identity) are importable by other modules. **They must never import `internal/`** (Go would allow it inside this module but break every external consumer) — the boundary is worth re-checking when adding code there. `metaclient` deliberately has no metrics dependency: the agent installs `Client.Observe` to bump `obs.MetadataRequests`.

## Architecture

Two binaries: `cmd/kubescrape` (metadata service) and `cmd/kubescrape-agent` (per-node DaemonSet: log tailer + Prometheus scraper exporting OTLP). Both live in one image; the DaemonSet overrides `command: ["/kubescrape-agent"]`.

### Metadata service

`cmd/kubescrape` serves Kubernetes pod/container metadata over HTTP. Everything is driven by one initial LIST + WATCH per resource (client-go informers); there is **no per-request API-server traffic**.

Data flow:

- **Pod informer** (full objects) → `internal/store.Store`: pods keyed by UID, with two derived indexes — normalized container-ID → entry, and nodeName → pods. `managedFields` are stripped by an informer transform before caching.
- **Service informer** (full objects) → `internal/services.Index`: selector-based matching of Services to pods, for service-annotation scrape discovery.
- **Metadata-only informers** (`PartialObjectMetadata`, cheap) for ReplicaSets, Deployments, Jobs, CronJobs, Namespaces, Nodes → `internal/owners.Resolver`. Owner chains and namespace metadata are resolved **lazily at request time** from these caches, never stored on the pod record. Adding a new owner kind means extending `owners.AllGVRs` + `kindGVR` and the ClusterRole in `deploy/kubernetes.yaml`.
- **ServiceMonitor CRDs** (opt-in `-servicemonitors`, dynamic informer over unstructured) → `internal/servicemonitors.Index`. A discovery pre-check disables the feature gracefully when the CRD group is absent. Targets resolve monitor → Services (label selector over `services.All(m.ServiceNamespaces())`) → pods (the existing service selector matching), so scraping stays node-local; `scrape.MonitorTargets` handles endpoint `port` (service-port name) / `targetPort` (pod port) / `path` / `scheme` and stamps `Source: "servicemonitor"` + `Monitor: "ns/name"`; an endpoint naming NEITHER port nor targetPort resolves to nothing (an empty `port` must not match a Service's unnamed port by `""==""` and fabricate a phantom target, as prometheus-operator emits no config for it). Per-endpoint auth/relabelings/intervals are deliberately not interpreted.
- **Events exporter** (opt-in `-events`, `internal/events`): the core Events informer feeds a channel-queued batcher exporting OTLP log records via the shared `otlpexport` client (flags `-otlp-*` on the service). Initial-list history is skipped by timestamp; count bumps re-emit; pod events are enriched from the store (UID cross-checked) AND owner-resolved via `Config.Owners` (the store never resolves the chain — without it `attrs.ServiceName` falls back to the POD name, so a workload's events would not share a `service.name` with that same pod's logs), finishing with `attrs.Identity`; delivery is best-effort (queue-full drops count `obs.EventsDropped` with a rate-limited warn, no retries; shutdown flushes the pending batch on a background timeout).
- `internal/server` composes the above per request; `internal/scrape` is pure functions deriving targets from `prometheus.io/*` annotations and ServiceMonitor endpoints; `pkg/kubemeta` holds the JSON model and the corev1.Pod → model conversion (`kubeconvert.FromPod`). Pods are also resolvable by UID via `GET /v1/pod-uids/{uid}` (`store.GetPodByUID`, tombstone-aware) and by pod IP via `GET /v1/pod-ips/{ip}` (`store.GetPodByIP`, deliberately NOT tombstone-aware — IPs recycle; hostNetwork and Succeeded/Failed pods excluded, a finished pod's retained status.podIP must never shadow the live owner; a TERMINATING pod — `record.terminating`, set from `p.DeletionTimestamp` — yields its IP claim to a live incumbent, since a drained pod keeps phase Running and still reports its now-recycled PodIP, while every live pod claims (last-write-wins), INCLUDING a late-scheduled older pod legitimately taking a freed IP. Creation time was the wrong signal — a pod's age says nothing about who currently holds the address) — the agent's OTLP-ingest enricher uses these.
- **Self-metrics are OTLP-pushed, not scraped**: `internal/obs` defines every internal metric on `metrics.Registry` (`internal/metrics/registry.go` — a prometheus-shaped API over the same `series` storage; no expiry/cardinality caps, `GaugeFunc` evaluated at export; `WithLabelValues` caches typed wrappers per label-value key, so hot-path counter bumps are zero-alloc). Both binaries run `obs.Registry.Run` on `-self-metrics-interval` (default 1m, 0 disables) with their own resource identity (agent: `service.name=kubescrape-agent`+node; service: `service.name=kubescrape`+hostname). `GET /metrics` (both binaries) serves ONLY Go runtime/process metrics (`obs.RuntimeHandler`, a dedicated prometheus registry with the Go+process collectors) — `kubescrape_*` metrics are never exposed there. New failure paths should count into an existing `obs` metric or add one there; tests read values via the `Value()` accessor.
- **HTTP metadata caching**: the service stamps metadata 200s (container/pod/pod-uid/node — deliberately NOT pod-ips, whose index exists for recycled-IP immediacy) with `Cache-Control: max-age` + `ETag` (`server.writeCached`, TTL `-metadata-cache-ttl`, default 10s, 0 disables); `metaclient` caches by URL (stripping the transient `?wait=`), serving fresh entries locally and revalidating stale ones with `If-None-Match` → 304. This cuts repeat lookups on the concurrent ingest and cadvisor paths (`obs.MetadataRequests{cached,not_modified}`). The client cache is mutex-guarded (`metaclient` is shared across concurrent ingest handlers) and stores the DECODED value alongside the body: hits and 304s shallow-copy it out instead of re-unmarshaling (maps/slices shared under the store's treat-as-immutable contract), and an undecodable 200 is an error and never cached (like 404s). `Client.now` is injectable for tests.

### Invariants the API guarantees (user requirements — do not regress)

- **Container lookups block per container ID.** An unknown ID registers a waiter channel keyed by that exact ID; the pod-upsert path wakes only those waiters. Multiple concurrent requests for the same ID are all released. Waiting is bounded by `-wait-timeout` (client can shorten via `?wait=`, never lengthen). Requests may legitimately arrive ~1s before the kubelet posts the container ID to the API server — this wait covers that gap.
- **Deleted pods stay resolvable via the container endpoint for `-cache-ttl`** (tombstone: `expireAt` + `deletedAt` stamped, swept periodically). Container IDs replaced by restarts get the same TTL; `lastState.terminated` IDs are indexed while the pod lives.
- **Scrape targets never include deleted pods**: `DeletePod` removes the pod from the node index immediately, before tombstoning. `Succeeded`/`Failed` pods are excluded from targets but remain resolvable by container ID.
- **`prometheus.io/port` is a comma-separated list** (numbers and/or port names) on both pods and Services; service ports translate to pod ports via `targetPort`. Duplicate endpoints reachable via both pod and service annotations are deduped by URL (pod source wins).

### Concurrency model

One `sync.RWMutex` in `store.Store` (plus one in `services.Index`) guards everything; there are no goroutines besides HTTP handlers, the informer callbacks, and the sweeper (`Store.Run`). Stored `kubemeta.Pod` values are never mutated in place — upserts replace whole records — so returning shallow copies under RLock is safe. `FromPod` deep-copies everything it takes from informer objects; never retain or mutate an informer-owned object.

Store tests use an injectable clock (`s.now`); follow that pattern for any time-dependent behavior instead of sleeping.

### Node agent (`internal/agent/...`)

The agent talks only to the metadata service (`metaclient`) and an OTLP gRPC collector (`otlpexport`, built on `go.opentelemetry.io/collector/pdata`) — no Kubernetes API, no RBAC (except `nodes/metrics` for the kubelet scrapes). Resource-attribute building lives in `agent/attrs`: `attrs.Builder` (nil = defaults) runs defaults → static → config templates (funcs: env/coalesce/default/regexMatch/regexReplace; context: Node/Pod/Container/Service) → `attrs.Filter`, driven by the config's `resourceAttributes` section + `-resource-attrs-static`/`-enable`/`-disable`; `attrs.Builders` holds one per pipeline (logs/targets/cadvisor/node/journal/ingest) merged from the section's `pipelines:` subsection. The defaults set `k8s.pod.ip` (from `Pod.PodIP`) as a resource attribute and derive `service.name` via `attrs.ServiceName` (workload owner → pod name), then finish with `attrs.Identity(res)`, which reads whatever identity attrs are on the resource to derive `service.namespace` (= k8s namespace) and `service.instance.id` (fallback: container.id, pod-uid[/container], namespace/pod[/container], node) for Mimir job/instance — neither overwritten if a template set it; it works for pod/container/node AND pre-populated (KSM splitter, ingest, log-metric) resources. **Instance-collision avoidance** (`attrs.PrefixInstance`, cmb-alloy `instance_prefix`): an exporter that DESCRIBES other objects (cadvisor, a KSM splitter) shares the described pod's `service.name`/namespace, so its `service.instance.id` would collide with the pod's own self-scraped metrics (same job+instance, different resource attrs → flapping target_info). The config's `instancePrefix` prepends `prefix-` to the derived instance; it defaults per pipeline via `defaultInstancePrefix` (cadvisor → `"cadvisor"`, others none), precedence **explicit pipeline section > built-in default > top-level base** (so a global base can't strip cadvisor's protection); split rules default it to the describing target's `service.name` (`splitBatcher.defaultPrefix`), overridable per rule (`SplitRule.InstancePrefix`, `""` disables). Split rules also take `dropLabels` (anchored name regex, labels omitted from data points in `putSplitLabels`) and `attributes` (set-if-absent resource fallbacks applied after `Build` in `fillSplitResource`); several groupBy labels may map to one attribute — applied in sorted label order, non-empty overwrites, giving a deterministic coalesce. On KSM/split resources the rule's `datapointAttributes` (config `datapointAttributes`, default `["k8s.node.name"]`, `defaultDatapointAttrs`) are moved off the resource onto the data points (cmb-alloy placement — a described object's node is a property of the object, not the exporter's identity): `splitBatcher.route` removes them from each split resource into `dpAttrs`, and `putSplitLabels` stamps them on every data point (regular self/scrape/cadvisor/node resources keep them as resource attributes). `[]` keeps everything on the resource. It is invoked at every point a final resource is built — tailer metadata resolution, both single-resource batchers, the cadvisor per-pod resources and the splitter resources (which may pre-put identity attrs before Build). New resource-producing code paths must call `Attrs.Build`. `.Node` metadata comes from `GET /v1/nodes/{name}/metadata` via the agent's `startNodeInfo` provider.
- **Series filtering & splitting** (`agent/promscrape/filter.go`, `split.go`, config `metrics` section): ordered first-match-wins keep/drop rules per pipeline applied in `parseAndExport` before conversion (so partial histogram families are possible, as with Prometheus relabeling); splitters route KSM-style targets into per-object resources via `splitBatcher` (groupBy label → attribute mapping + optional metadata enrichment; requires `Kubelet.Meta` even without kubelet scrapes; the shared `podCache` is mutex-guarded because splitters run on concurrent scrape goroutines).
- **Exporter** (`agent/otlpexport`): gRPC or OTLP/HTTP (protobuf POST to /v1/logs, /v1/metrics), bearer token file re-read per minute, TLS CA/skip-verify, bounded retries for metrics only — log retries/rewind belong to the tailer. **Size-bounded sending** (`pkg/otlpsplit` + `Config.MaxSendBytes`, `-otlp-max-send-bytes`, default ≈3.75 MiB under the 4 MiB gRPC limit): every send measures the exact proto size (`plog/pmetric/ptrace ProtoMarshaler.*Size`, attributes+framing included, not just bodies) and splits an over-cap payload into parts each within the cap, preserving resource/scope grouping (`otlpsplit.Logs`/`Metrics`/`Traces`; an over-cap resource splits by record/metric/span, a single leaf over the cap goes alone). This is the ONE hard guarantee against wholesale rejection for producers that batch by count (journald, tailer) rather than pre-chunking (promscrape, the ingest batcher already stay under the cap and never trip it). A part-send failure returns immediately and the caller retries the whole payload — at-least-once tolerates the re-sent parts, and nothing commits until every part lands. A non-empty input never yields zero parts (an over-cap record-less resource is sent whole, rejected+counted at the collector, never silently reported delivered); an empty scope's identity is preserved across a big-resource split. Payloads are gzip-compressed by default (`Config.Compression`, `-otlp-compression=gzip|none`) via klauspost/compress: `gzip.go` registers a klauspost-backed gRPC "gzip" codec (do NOT import grpc/encoding/gzip too — last init wins) and gzips HTTP bodies with pooled writers into `github.com/JohanLindvall/bufpool` buffers — the pooled buffer IS the request body (the transport's Close recycles it; never set `req.GetBody`, a replay would read recycled memory).
- **Disk buffer** (`pkg/spool` + `otlpexport.Buffered`, opt-in `-buffer-dir`/`-buffer-max-bytes`): a Fluent-Bit-style filesystem write-ahead buffer. `spool` is a durable FIFO byte queue — rotating segment files (`<seq>.seg`), each opening with an 8-byte header (magic + big-endian format version, currently 1) followed by frames of `len(4) + xxhash64(8) + payload`; the checksum covers the LENGTH as well as the payload, so a flipped length byte is caught instead of mis-framing the rest of the segment. A segment whose header is missing or names an unknown version is DROPPED at open (the spool is a transient buffer — no legacy readers; to change the framing, bump `formatVersion`, extend `frameHeaderLen`/`knownVersions` and the frame paths — the version is per segment, so both coexist); a DAMAGED (missing-magic) drop is surfaced as one `ErrCorrupt` from the next `Pop` so the loss is counted, an unknown-*version* drop stays silent (a deliberate format change). **Segment seqs are monotonic for the spool's life**: after every segment is dropped, numbering resumes at `max(1, cursorSeq+1)`, never back at 1 — otherwise a fresh segment could reuse the exact seq a persisted cursor still names and be mistaken for the consumed one, deleting live records. A cursor naming a seq newer than any segment on disk (foreign/hand-corrupted) is discarded and the backlog redelivered from the oldest segment, never trimmed to it. `fsync` per Append; on reopen the tail is truncated only at STRUCTURAL breaks (a crash-interrupted frame), never at a checksum failure — a whole-but-damaged frame stays and `Pop` drops just it (`ErrCorrupt`), so one bad byte costs one record rather than everything behind it. A checksummed 24-byte in-place `cursor` (torn → redeliver from the oldest segment); consumed segments retire in `Pop` as well as `commit`. `Pop` surfaces read errors — a vanished head segment or `ErrCorrupt` data is skipped and counted `obs.BufferReadErrors{lost=true}` rather than wedging the drain; `MaxBytes` caps the *undelivered backlog* (`backlog = Σsegsize − readOff`), returning `ErrFull` (tailer rewinds = back-pressure). `Buffered` wraps the client with one spool per signal (generic `sink[T]` over plog/pmetric), implements the shared `otlpexport.Exporter` (ExportLogs+ExportMetrics), enqueues on Export and drains in `Run` with retries (single-attempt inner sends; the backoff persists across cycles to the 30s cap) — commit only after the collector acks (at-least-once across restarts). `IsPermanent` classifies definitive rejections (dropped + counted); auth failures and 404 are deliberately transient — a rotating token or a collector rollout must not drain the backlog into drops. A batch whose TRANSIENT rejection is in fact permanent for that payload (an over-limit message comes back as `ResourceExhausted`) would otherwise circle the queue forever: it is dropped + counted after `maxDrainCycles` accountable failed cycles (`sink.stuckTooLong`, keyed by payload hash). A cycle is accountable only when BOTH hold: the collector delivered some OTHER batch while THIS one was stuck (`sawProgress` — armed by `s.delivered` advancing between the batch's own failed cycles, so deliveries banked before it got stuck, or during a recovery it merely sat behind in the queue, never count), AND the cycle's final error was a collector RESPONSE (`respondedError`: an HTTP status or a non-Unavailable/DeadlineExceeded/Canceled gRPC status) rather than a transport failure — so a blip delivery followed by a resumed outage spends nothing, and a collector rejecting EVERYTHING (a memory-limiter overload) never arms. Only a live collector repeatedly refusing THIS payload while accepting others is poison evidence. A recovered stuck payload is `forget`-ten on success (and on a later permanent rejection) so the tracking map cannot leak. When enabled, main routes ALL consumers (tailer, journald, scrape, log-metrics, ingest) through the single `out` value; the buffer's inner is the raw client. Physical disk can exceed the cap by up to one 8 MiB segment (delivered-but-unreclaimed prefix).

The synthetic `__line__` key (`logline.LineKey`) returns the whole raw line for content selectors/labels. Counter/gauge/histogram/summary (`seriesKind`); gauge has an `action` (set/inc/dec/add/sub + windowed aggregations min/max/avg/first/sum/count/stddev/range/delta, `series.record`+`series.snapshot`+`series.aggregateValue`; - **Tailer performance** (`bench_test.go`): the multiline package (>= v0.0.4) prefilters its start-state regexes with literals derived from the patterns — lines containing no literal skip all start regexes (~12x less per-line CPU). If a pattern change makes the literal set unprovable the matcher silently falls back to full regex evaluation (correct, slower); `TestPrefilterEnabled` (via `patterns.StartLiterals()`) is the alarm. `feedLine` parses the CRI header once and feeds the result to `cri.AddParsed` (multiline >= v0.0.6; the stage's internals never re-parse — the single `cri.Parse` per line is feedLine's), uses precomputed `file.keyStdout`/`keyStderr` (no per-line key concat), and the ledger's offset accounting lives in per-stream `streamState` structs reached via pointers cached on the file (`stStdout`/`stStderr`/`stPlain`, re-derived after every `reset()`) — no map operations per line. The `streamState` fifo of buffered offset ranges is consumed by advancing `fifoHead` (`live`/`push`/`pop`), never by re-slicing: re-slicing walked the backing array forward until its capacity ran out and then, since the steady state is one line pushed and one popped, reallocated a one-element array for EVERY line — that was the tailer's entire per-line allocation. **The line path is now zero-alloc** (BenchmarkIngestLine: 0 allocs/op; keep it there). BenchmarkIngestLine/BenchmarkIngestFlush track the per-line budget (~0.2µs pipeline, ~0.8µs with enrich).
- **Tailer** (`agent/tailer`): single sweep goroutine over the configured log sources (see the Log sources bullet; default `/var/log/containers`). Log lines flow through the two-stage `github.com/JohanLindvall/multiline` pipeline, per file: `multiline/cri` parses the CRI format and rejoins P/F fragment runs (non-CRI lines pass through rather than being dropped), then `multiline` joins stack traces. Offset mapping relies on stage emission being synchronous inside `Add`/`Flush*`: `lastEnd[key]` is exact because the line completing a run is always the line currently being fed, and `Entry.Lines` (which counts even limit-dropped lines) pops the per-key FIFO of logical-line offset ranges. At-least-once invariants: exports happen inline in the sweep with retries; per-file offsets are committed only after a successful export and never past lines still buffered in either stage (`watermark` = min of `runStart` and FIFO heads); on failure files are rewound to the committed offset and the pipeline is discarded unemitted (the lines get re-read); committed offsets are checkpointed to disk via the positions store (persisted immediately when a new file is discovered — a crash before a file's first entry would otherwise skip it as history on restart; where a checkpoint-less file starts is `-logs-unknown-files`: auto/end/start, auto = read from start when the store already has entries). A file's metadata is resolved (with server-side wait, 10s failure backoff) before its data is consumed — nothing is read until it can be attributed; a file deleted before it ever resolves is released with the loss COUNTED (`obs.LogUnresolvedLost`), since nothing was read and nothing can be. Idle files close their fd after `-logs-idle-close` (default 0 = never; the file stays tracked and reopens with identity re-verified on activity) — steady-state fds are one per active file, not per tracked file. Rotation: identity = inode + head fingerprint (FNV of first `-logs-fingerprint-bytes`; catches inode reuse AND copytruncate — the post-read `read == 0` check catches a same-or-smaller rewrite, and a PRE-read fingerprint re-verify in `readFile` (whenever the file's mtime moved since the last read) catches a copytruncate whose replacement content is LONGER than the read offset. fingerprints EXTEND (shared `extendFingerprint`, called from `saveCheckpoints` AND from readFile after a successful read — without the read-path call a store-less deployment never extends and a file first opened at size 0 keeps the matches-anything empty fp forever) only while the head already hashed still matches — re-hashing unconditionally would adopt a copytruncate replacement's head and blind both guards for every sub-`FingerprintBytes` file; the truncation DECISION in readFile uses the PRE-settle readPos (settle can rewind it and re-evaluating against the shrunken value missed an in-place rewrite sized between committed and readPos); rename rotation drains the old fd before switching; deletion drains before drop. **Segment-qualified offsets** (the decomposition docs/tailer-comparison.md describes — DONE): every buffered/emitted offset is a `pos{seg, off}` where `seg` is a per-file monotonic incarnation id (the live file is the TAIL segment; each rename rotation closes the tail into a `segment` record `{id, inode, fp, committed, to, fd}` on `file.segments` and issues a fresh tail id). A segment's `to` is the last FED line boundary (max `streamState.lastEnd` at the tail), NOT readPos: trailing bytes that never entered the pipeline (torn fragment, blank line, rate-dropped or oversize-discarded lines) can never produce a committing entry, and a `to` covering them pinned the segment below retirement forever. An UNRECOVERABLE segment (findRotated miss, file pruned) is counted (`obs.LogPrefixLost`) AND retired — keeping the record wedges retirement. `segmentsFed` asserts every recorded segment's owed lines are live (pipeline or batch); rotation preserves the incoming value (`wasFed`, vacuously true with no prior segments) rather than overclaiming — a mid-drain rewind purges older segments' re-fed lines, and marking them fed stranded an older rotation's lines until restart. A mid-drain flush FAILURE aborts the drain (`drainReader` returns false — the rewind seeks the very fd being drained back, and continuing was a hot loop burning export attempts on the single sweep goroutine); the rotation is retried next sweep with the fd held. Cross-rotation ambiguity is inexpressible by construction: an entry commits against `end.seg`'s record — the tail candidate advances the file checkpoint, an old segment's candidate advances that segment's `committed`, a segment retires (`retire`: fd closed, checkpoint entry gone) when its whole range commits, and a DEAD id (truncated-away incarnation, or the archive-replaced restart's `newTail`) resolves to nothing. This replaced the old rotation-generation protocol (`gen` stamping + apply-time checks) and the `reanchor` offset rewrite wholesale. **Multi-line groups (CRI P/F runs and stack traces) join across a rename rotation**: if data is still buffered after draining the old inode, `reopen` carries the pipeline — buffered items keep their old-segment positions untouched — and the fresh tail id disambiguates the new inode's bytes. Zero-loss across a crash, including multiple rotations: incomplete segments are checkpointed (`positions.Prefix` list, oldest first; identity = inode+fp; `From` is the segment's commit PROGRESS, so a restart re-reads only the owed `[From,To)` remainder), and `reopen` persists the checkpoint IMMEDIATELY when a segment is recorded — the 10s cadence alone left a crash window that lost the rotated tail outright (discovery persists immediately for the same reason). Each segment RETAINS the rotated inode's fd but the retained fds are CAPPED (`maxCarriedFds`, oldest kept — they prune first) so a long outage across many rotations cannot exhaust fds or pin unbounded disk; `reopen` NEVER resets `segments` (a later rotation must not drop earlier uncommitted ones); on restart (or after a rewind that discarded the buffered group, tracked by `segmentsFed`) `feedSegments` re-reads each incomplete segment's owed range UNDER ITS OWN id (`ledger.feeding`) — `findRotated` (by inode+fp in the resolved target dir) is the post-RESTART fallback (no fd survives a restart); when it too fails the loss is counted (`obs.LogPrefixLost`) and the segment retired. A rotation that happened while the agent was DOWN is recovered too: `initFile` sees the checkpointed identity no longer matching the path and synthesizes an OPEN-ENDED segment (`to = -1`; feedSegments reads the rotated file to EOF and pins the range, or counts + retires when pruned). Caveat: the join only spans rotations the tailer actually **observed** while running (it follows the symlink, not every intermediate rotated file — a rotation so fast the intermediate inode is never opened loses that segment, pre-existing). Truncation/copytruncate still flush-and-reset (content replaced, joining meaningless). Sweeps are fsnotify-driven (watch the symlink dir + resolved target dirs; files directly in the watched dir are a test-only special case) with the poll ticker as fallback. **Log rules** (`Config.Rules` = `metrics.LineFilter`, config `logs.rules`): ordered keep/drop/sample evaluated in flush AFTER enrich (synthetic `__severity__` via `metricResolver.ruleLookup`) and AFTER LogMetrics (metrics see every line); with rules set, records build in a one-record scratch slice and `MoveAndAppendTo` into the batch only when kept; dropped records still advance offsets (an all-dropped batch commits without exporting) and count into `obs.LogRulesDropped`. **Rate limiting** (`Config.RateLimit/RateBurst/RateDrop`, flags `-logs-rate-*`): per-file token bucket in `allowLine`/`consume`; pause mode sets `file.limited` (readFile/readArchive stop reading, pending retried next sweep — for archives BEFORE the archiveDone short-circuit, or an EOF-paused tail wedges), drop mode discards lines; rotation drains bypass the limiter (`consume(..., unlimited=true)`) or the rotated remainder would be lost. Pause mode's no-loss contract holds across EVERY restart path: `reopen` and the archive restart salvage complete pending lines (unlimited) before the reset, and the shared `file.restartAt` clears `limited`/`discarding` with pending everywhere (openArchive included — a wiped pending with `limited` still set wedged the file forever, and the archiveReplaced block once drifted from reopen by omitting exactly these resets). An unterminated fragment at a RENAME rotation can never complete and is dropped counted (`obs.LogTornFinalLines`); an oversized unterminated line (> MaxEntryBytes+4096) discards its prefix AND its eventual remainder (`file.discarding`, `obs.LogOversizedDropped`) — the mid-line suffix must not export as a record. **Status** (`status.go`): `publishStatus` (housekeeping, `statusEvery` 10s) snapshots per-file positions/lag into an atomic pointer read by `Tailer.Status()` → agent `GET /debug/tailer`; gauges `obs.LogLagBytes{,Total}`; compressed files report `Lag = ReadPos − Committed` (Size is compressed bytes, offsets are decompressed — their difference is meaningless). **Withheld commits**: a record exported while ANOTHER stream's multi-line group is buffered has its commit clamped to the build-time watermark (a candidate in a segment NEWER than the watermark's commits nothing, same segment clamps to its offset, older segments are unconstrained); the unclamped high position is remembered (`file.exportedHigh`, via `batchInfo.highs`) and re-offered at the next flush touching the file — without it `committed` froze below `readPos` forever (restart re-reads the tail, idle-close never releases the fd, lag gauges show phantom backlog). Exports are synchronous in the sweep (`flush.go`: `batchInfo` carries build-time per-segment candidates to `commitBatch`/`failBatch`); the former opt-in pipelined-export mode was REMOVED — its settle protocol cost more in invariants than the overlap earned, and `-buffer-dir` provides the same decoupling with at-least-once durability (docs/tailer-comparison.md).
- **Log sources** (`agent/tailer/sources.go`, config `logs` section): the tailer discovers files by `[]Source` (doublestar include/exclude globs, `**` supported) rather than a single dir; empty config = one default containerd source over `-log-dir`. Each source is `containerd` (CRI filename→containerID→metadata→CRI parse) or plain (arbitrary files: no metadata lookup, resource attrs = builder node attrs + the source's static `attributes`, `service.name` defaulting to the source name; per-source `multiline` override). A file is claimed by the first matching source — INCLUDING a containerd source that skips it (excluded namespace / unparseable CRI name): `ExcludeNamespaces` is global config (the observability feedback-loop guard) and a later catch-all source must not resurrect the file. Non-regular files (FIFOs, sockets) are never tracked — open/read on a FIFO blocks forever and would wedge the single sweep goroutine node-wide. A non-gzip file matched by a compressed source is quarantined under its current identity (archiveDone + size/mtime; a valid rewrite recovers on its own) instead of warn-looping every sweep. Plain files bypass stage-1 (CRI) — `feedPlainLine` feeds the trace stage directly under a single `plainKey` so **the identical rotation / offset / cross-rotation-multiline machinery applies** (`file.source` selects the path; `newPipeline`/`feedLine`/`resolveMetadata`/`stopPipeline` branch on `source.containerd`, guarding nil `criStage`). The watcher watches every source's fixed include base dirs (`scanDirs`). Per-source `compressed` (gzip): compressed files use a separate one-shot `readArchive`/`openArchive` path (offsets in *decompressed* space; gzip isn't seekable so resume re-decompresses and `io.CopyN`-discards the committed prefix; read whole, not skipped-to-end; `rewind`/`drop` branch on `f.compressed`). The archive's fd is RETAINED past EOF (`archiveEOF`) and across a rewind while its data is uncommitted (`closeArchiveReader` closes only the gzip reader; `openArchive` re-decompresses from the retained fd) — an export can fail and logrotate can prune the `.gz` before the retry, and the fd is then the only handle to it; it is released once `committed >= readPos`. An archive REWRITTEN in place (same inode, new content) is detected by `archiveReplaced` (fingerprint of the held fd, checked each `readArchive`) and restarts the file at zero — without it the reuse path would resume at a stale decompressed offset; and the EOF path only stamps `archiveDone` when the path still names the inode we read, so a replacement is not mistaken for consumed — the incremental containerd/plain path is untouched when `f.compressed` is false. Caveat: `consume` drops empty physical lines, so multi-line formats needing a blank separator (Go panics) don't join for plain files (CRI's blank is a non-empty `…F ` line); indentation-based traces (Python/Java/.NET) join fine. Non-UTF-8 transcoding was intentionally dropped (Linux-only target) — lines are UTF-8/passthrough. Opt-in `-logs-file-attributes` (`Config.FileAttributes`) stamps `log.file.name` (basename) + `log.file.position` on every record; `entry.start` is the record's start offset (first fragment's, threaded through cri as the entry data / `items[0].start` for trace groups), distinct from `entry.offset` (the post-record commit offset).
- **Scraper** (`agent/promscrape`): hand-written streaming parser for the Prometheus text format, classic + OpenMetrics (constant memory; do NOT replace it with `expfmt`, which buffers whole metric families — the 100k-series requirement is why it exists). The parser classifies samples into roles (`parser.go`); `convert.go` groups histogram/summary component series per family and label set into proper OTLP Histogram/Summary points — grouping state lives only for the current family, so memory stays bounded by the largest family, not the scrape. Exemplars are opt-in (`Config.Exemplars` / `-scrape-exemplars`): enabling them switches the Accept header to OpenMetrics; the parse mode is always detected from the response Content-Type (OM uses float-second timestamps, classic integer milliseconds). Conversion chunks per export via the `chunker`/`sink` interfaces at whichever of `BatchPoints` data points (10k) or `BatchBytes` estimated bytes (`defaultBatchBytes`, 3 MiB) comes first. **The byte bound is not optional**: a collector's default gRPC receive limit is 4 MiB and applies to the DECOMPRESSED message, and a point count does not bound bytes (10k points of a label-rich family marshal past 5 MiB → every export of that target is rejected = total metric loss). Each sink tracks the estimate (`numberBytes`/`histBytes`/`summBytes` for points, `resourceBytes` + `metricOverheadBytes` charged where a resource/metric is CREATED) and the converter re-checks it BETWEEN the points of a flushing histogram/summary family (`converter.emit`), so one huge family cannot overshoot either bound. **Counting only data points is not enough**: the split and cadvisor batchers emit one ResourceMetrics per DESCRIBED OBJECT, each with a full enriched attribute set plus its own copy of every metric descriptor — a 12k-pod KSM split flushed at an estimated 3 MiB and encoded to 6.7 MiB, i.e. straight past the limit the estimate exists to respect. An aborted parse (sample limit, truncated body, read timeout mid-body) still exports what was converted before the abort — every kind here is cumulative, so a partial scrape costs nothing but the missing series. **Hot-path allocation discipline** (bench_test.go guards it; ~4→0 allocs/sample): the parser interns metric/label names and label values per scrape (`internName`/`internValue`, capped — `maxInterned*`; `map[string(b)]` lookups don't allocate) and short-circuits even the intern probes with last-seen memcmp caches (`lastMetric`, per-position `lastKV` — consecutive lines of a family are near-identical; `string(b) == s` doesn't allocate), whitespace is skipped with `skipSpaceTab` (NOT `bytes.TrimLeft`, which builds an ASCII set per call), `converter.labelKey` fingerprints into a reused buffer (`keyBuf`/`keyLbl`) with a keyed `map[string(buf)]` lookup, the batcher's `metricByName`/`remember` cache the last family, and `parseAndExport` filters through `MetricFilter.session()` (per-scrape name→rule-bitmask memo plus a per-(matcher,value) label-regex memo — label values repeat heavily; the shared filter stays immutable). The last-seen caches also skip the byte scans: `parseSample`/`parseLabels` match the previous line's name plus terminator in one memcmp, and `parseQuoted` finds the closing quote with `bytes.IndexByte` — keep new per-sample code free of per-call closures and `fmt`. Per-scrape state is pooled across scrapes: `getParser`/`putParser` reuse the parser + its `bufio.Reader` (family `types` cleared per exposition, intern maps kept warm with cap-based clears), and the converter recycles histogram/summary accumulators via `histFree`/`summFree` — a 10k-series scrape parses in ~3 allocs. Prometheus 3.x quoted UTF-8 metric names (`{"my.metric",code="200"} 1`) are NOT supported: such lines are counted malformed and skipped (bare dotted/UTF-8 names without the quoted-brace syntax parse fine); a target migrating to the quoted syntax loses those series with `obs.ScrapeMalformed` as the only signal.
- **Kubelet scrapes** (`agent/promscrape/cadvisor.go`; cgroup-path parsing in `pkg/cgroupid`): with `-kubelet-endpoint` the agent scrapes `/metrics/cadvisor` and `/metrics`, bearer-authenticated with the mounted ServiceAccount token (the agent's ServiceAccount exists ONLY for `nodes/metrics`). cadvisor routing (`cadvisorBatcher`) keys resources by the cgroup path in the `id` label (`cgroupid.Identity` parses both cgroupfs and systemd layouts): container ID → exact incarnation via `/v1/containers/{id}` (non-blocking); pod-level series (e.g. `container_network_*`, which have NO container id) → `/v1/pods/{ns}/{name}` cross-checked against the cgroup pod UID; sandbox rows (`container="POD"`) fold into the pod resource (their cgroup names the pause container — never look that ID up; `ident.image` is cleared too, it names the pause image). On pod/container-identified rows (`ident.podScoped()`) the redundant `id`/`name`/`image` labels are elided from data points (`redundantOnPodRow` in `putFilteredLabels`; unresolved resources keep the image as `container.image.name`); rollup rows keep `id` — their only distinguisher on the shared node resource. Lookups share a 1-minute TTL cache. `-cadvisor-rollups=false` drops above-pod-level cgroup series AND pod-level rows of container-scoped families; `podScopedFamily` (container_network_*) is the keep-list of families with no per-container breakdown. All pipelines toggle independently: `-logs`, `-metrics`, `-cadvisor`, `-node-metrics`, plus opt-in `-journald`. `Config.HealthMetrics` exports per-target `up`/`scrape_duration_seconds`/`scrape_samples_scraped` gauges after each cycle (`scrapeOutcome` collected in `cycle()`, `exportHealth()`).
- **journald** (`agent/journald`, opt-in `-journald`, **cgo**): reads the journal natively via `github.com/coreos/go-systemd/v22/sdjournal` (libsystemd) — no journalctl subprocess. One resource per unit (identity attrs pre-put, then `Attrs.Build` with the `journal` pipeline). The reader is decoupled behind a `source` interface (`next`/`close`) opened by `Reader.open` (default `openJournal` in `source_sdjournal.go`; tests inject a `fakeSource`), so `journald.go` (batch/convert/cursor/flush) has no cgo and is fully unit-tested. A MESSAGE over `MaxEntryBytes` is truncated on a rune boundary and the record carries `log.truncated`+`log.original_length`, counted `obs.JournalTruncated` (never a silent cut). sdjournal seek: resume via `SeekCursor` then skip the (still-present) cursor entry once; tail via `SeekTail`+`Previous`; `next` loops `Next`/`Wait(1s)` honoring ctx. At-least-once via the cursor: committed only after successful export; on failure/reader error the source is closed and reopened from the committed cursor with backoff. **While NO cursor has been committed yet** (first run, or no positions store) the restart path cannot recover the buffered entries — a reopen with an empty cursor seeks to the journal TAIL — so `Run` keeps retrying the pending batch's export (with backoff) instead of reopening; the journal itself is the buffer, and the first successful export makes the normal re-read semantics apply. Do not "simplify" that back into an unconditional restart. Because the agent links libsystemd it is built **CGO_ENABLED=1** (the metadata service stays CGO_ENABLED=0 static); the image is `distroless/base` + copied libsystemd (+ libcap/libgcrypt/libgpg-error/liblz4/liblzma/libzstd), not `distroless/static`.
- **Log enrichment** (`agent/logenrich`, flags `-logs-enrich`/`-journald-enrich`, both default true): `logenrich.Apply(lr, line)` runs `github.com/JohanLindvall/enrich` over each exported line in the tailer and journald `convert`/`flush` paths. Parsed timestamp overrides the CRI/journal time (ingest time stays in ObservedTimestamp), explicit level overrides severity (enrich's severity numbers ARE the OTLP numbers — cast directly), 32/16-hex IDs (dashes stripped, all-zero rejected) fill the trace/span fields, the rest become record attributes (`log.template`, `exception.*`, `azure.*`, `cloud.resource_id`). `exception.stacktrace` is suppressed for `Format == pattern` (a pattern-parsed trace is a verbatim slice of the body — attribute would double megabyte multiline entries; JSON-carried traces stay, and a substring check does NOT work: single-frame JSON traces contain no escapes and are substrings of the raw line). `Result.Format` feeds `obs.LogEnriched{format}` ("" → "none"). Body is never modified. New log-exporting paths should call `Apply` behind a config toggle.
- **Log-line attribute lifting** (`agent/logattrs`, config `logAttributes` section, shared by `-logs` and `-journald`): ordered rules map a JSON/logfmt key in the line to a resource/scope/log attribute. JSON is scanned once for all rule paths via `github.com/JohanLindvall/lightning/pkg/json` `GetPaths`+`DecodeAny` (dotted keys descend into nested objects); logfmt via the `logfmt` reader; do NOT reintroduce `encoding/json`. Resource/scope attributes participate in the flush grouping key (`logGrouper` in the tailer, the `unit\x01res\x01scope` key in journald `convert`) so records with different line-derived resource/scope attrs split into distinct ResourceLogs/ScopeLogs; log-target attrs go on the record. Typed: JSON numbers with no fraction → PutInt, else PutDouble; bool → PutBool.
- **Unified positions** (`agent/positions`, flag `-positions-file`): one JSON file holding both the tailer's per-file `LogPos` map and the journald cursor. The `Store` keeps both sections in memory and rewrites the whole doc atomically under a mutex, so the tailer and journald (different goroutines, independent sections) never clobber each other. It is the ONLY offset persistence (the legacy standalone `-checkpoint-file` was removed); the tailer's `checkpoint` type is a `positions.LogPos` alias. **journald has no standalone cursor file** — its cursor persists only through `positions.Store` (`journald.Config.Positions`); without a positions store it starts at the journal tail each run.
- **OTLP ingest** (`agent/otlpingest`, opt-in `-ingest`): a gRPC (`:4317`) + OTLP/HTTP-protobuf (`:4318`, `/v1/logs`+`/v1/metrics`+`/v1/traces`) receiver for telemetry apps push to the local agent. `Enricher` finds a container id (`container.id`/`k8s.container.id`, configurable) or pod uid (`k8s.pod.uid`) on a resource, resolves via `metaclient` (Container = exact incarnation, else `PodByUID`), and merges the `attrs.Builders.Ingest` k8s attributes **without overwriting** anything the sender set. Metrics mode (`-ingest-metrics-mode`): `resource` (id on the resource), `datapoint` (id per point → `split.go` regroups every point into per-object ResourceMetrics, all 5 metric types; points with no id fall back to the resource-level id), or `auto` (resource if every RM has an id, else split). In a split group the resolved identity **overwrites** the copied sender attributes (`overwriteAttrs`) rather than merging: the sender is authoritative about ITSELF, not about the other objects it describes, so its own `k8s.pod.name`/`service.name` would otherwise mislabel every split point. **Traces** (`-ingest-traces`, default true): `EnrichTraces` enriches ResourceSpans the same way; forwarded via `TracesExporter` (`otlpexport.Client.ExportTraces`; `Buffered` passes traces through UNBUFFERED — the pushing sender owns retry). **Peer-IP fallback** (`-ingest-peer-ip-fallback`, opt-in): a resource with NO id resolves via the connection's peer IP (`peer.go`: gRPC `peer.FromContext` / HTTP RemoteAddr → ctx; `Enricher.peerPod` → `metaclient.PodByIP` → `GET /v1/pod-ips/{ip}`; the store's `byPodIP` indexes LIVE, running-phase, non-hostNetwork pods only — deleted and Succeeded/Failed pods drop out immediately since pod IPs recycle); counts as `obs.Ingested{peer_ip}`; the splitter's ""-group applies the same fallback. **Batching** (`-ingest-batch-items`/`-ingest-batch-bytes`/`-ingest-batch-timeout`, `internal/agent/otlpbatch`): opt-in per-signal coalescing with collector-batch-processor semantics (enqueue acks the sender; bounded queue back-pressures with a retryable error; transient export failures retry with capped backoff — bounded by `maxDeliverAttempts`, then drop+count — while the full queue back-pressures; an oversized payload passes through unsplit; shutdown stops the server first, then drains) — pair with `-buffer-dir` for at-least-once. The shutdown drain gives each acked payload ONE attempt (the drain's ctx is already cancelled) and is bounded overall by `shutdownDrain` (10s): against a dead collector each attempt burns the exporter's full timeout, so an unbounded drain would outlive the pod's termination grace and the SIGKILL would take the rest of the backlog down silently — past the deadline the remainder is dropped and counted (`obs.IngestDropped`) instead. Caveat: batching acks TRACE senders too, trading their own retry for best-effort delivery (traces are never disk-buffered). Log bodies additionally run `logenrich.ApplyBody` (non-overwrite variant) when `-ingest-logs-enrich`. Counts into `obs.Ingested{enriched,unresolved,peer_ip}`. Uses the shared `otlpexport` client to forward.
- **Span metrics** (`agent/spanmetrics`, opt-in `-ingest-span-metrics`, requires `-ingest-traces`): RED metrics derived from ingested spans, following the OpenTelemetry spanmetrics conventions — a monotonic `traces.span.metrics.calls` counter, a `traces.span.metrics.size` byte counter, and a `traces.span.metrics.duration` histogram (seconds, with per-bucket trace-id **exemplars**), dimensioned by `service.name`/`span.name`/`span.kind`/`status.code` plus configurable extra span-then-resource attribute keys (`traceMetrics` config section: `namePrefix`/`buckets`/`dimensions`/`maxCardinality`/`exemplars`). Each span is aggregated INDEPENDENTLY, so a node-local agent (seeing only the spans pushed to its node) still produces correct per-service RED metrics — Prometheus sums the cumulative counters across agents. Service-graph edge metrics are deliberately NOT derived: pairing a request's client and server spans needs both, and they usually land on different nodes' agents, so a single agent never sees both halves. It plugs in as a `TracesExporter` **tap** (`Generator.Tap`) wrapping the RAW trace exporter, BELOW the ingest batcher: the server enriches spans in place BEFORE the export path runs, so the tap sees enriched spans (service.name populated); the tap forwards FIRST and aggregates only on success — a transient failure surfaces retryable to the sender, whose re-push must not double-count — and sitting under the batcher it runs on the batcher's own goroutine against a payload nothing else mutates (above it, Consume would race the batcher's merge). The batch is forwarded UNCHANGED. The generator is a **self-contained cumulative aggregator** (`spanSeries` map keyed by the length-prefixed dimension tuple), NOT `metrics.Registry` — exemplars are a histogram-data-point feature the Registry cannot express, and owning the aggregation also gives the size counter and OTLP units a single home. A **cardinality cap** (`maxCardinality`, default 20000) bounds the data-driven label sets: a new tuple over the cap is dropped and counted (`obs.SpanMetricsDropped`), existing tuples always keep reporting (cumulative). **Zero-alloc warm path**: `observe` builds the map key on a stack buffer and looks up via `map[string(key)]` (Go elides the copy for the read), materializing the dimension slice only when a genuinely new series is admitted — `BenchmarkConsume` is 0 allocs/op (`bench_test.go` guards it). Exemplars are one-per-latency-bucket, reset after each DELIVERED export (a failed send keeps them for the retry), and skip spans with no trace id; `exemplars: false` disables them. `spanSize` is a cheap allocation-free byte estimate (name + ids + attributes + events + links), not the exact proto size. `Consume` runs on concurrent ingest goroutines — the series map is mutex-guarded. Exported on `-ingest-span-metrics-interval` (default 1m) via the shared metrics exporter (`out`), under the agent's own resource identity (`service.name=kubescrape-agent`+node, `agentSelfResource`) — the described service is a data-point dimension (`service_name` a label, not the metric's job). Naming follows the OTel Collector connector, NOT Grafana Tempo's metrics-generator (`traces_spanmetrics_*`/`SPAN_KIND_SERVER`); the values differ from Tempo in metric name and kind/status casing.
- Tailer tests are timing-based: use `startTailer` (guarantees file creation happens after the initial scan — files present at startup are skipped to their end) and `tl.retryBackoff` to keep retry tests fast. A multi-line group that must survive across sweeps (e.g. straddling a rotation) needs **current** CRI timestamps on its lines (`timeNowCRI()`), not the usual hard-coded `2026-07-05`: the multiline stage's age-out (`FlushBefore`) runs against the real clock, so stale line timestamps flush the group before the continuation arrives.

### E2E in kind

`hack/otel-collector.yaml` deploys a contrib collector (debug exporter). Gotchas learned the hard way: with `verbosity: detailed` the collector's own stdout gets huge and kubelet rotates it within seconds (capture with `kubectl logs -f`, not `--since`); the agent must exclude the observability namespace (`-logs-exclude-namespaces`) or it feeds the collector its own output; after a collector rollout the agent's gRPC connection may keep delivering to the terminating pod for a while.

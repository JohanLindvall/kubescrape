## Commands

```sh
make build              # build to bin/kubescrape
make test               # go test ./...
go test ./internal/store/ -run TestWaitIsPerContainer -v   # single test
make vet fmt tidy
make lint               # golangci-lint v2 (auto-installed to GOPATH/bin; config .golangci.yml)
make image              # docker build; service is CGO_ENABLED=0 static, agent is CGO_ENABLED=1 (libsystemd/journald) on distroless/base
```

`CGO_ENABLED=1 go test -race ./...` works (gcc is available); run it for concurrency-touching changes (store, metaclient cache, otlpingest). The concurrency tests in `internal/store` and `TestEnricherConcurrent` are the targeted exercises.

### End-to-end verification (kind)

Changes to informer wiring, the store, or the HTTP API should be verified against a real cluster, not just unit tests:

```sh
make cluster-up         # 3-node kind cluster; downloads kind/kubectl into hack/bin; deploys hack/test-workloads.yaml
go run ./cmd/kubescrape # uses the kind kubeconfig context
make image && hack/bin/kind load docker-image ghcr.io/johanlindvall/kubescrape:latest --name kubescrape
kubectl apply -f deploy/kubernetes.yaml   # in-cluster deployment (namespace: monitoring)
make cluster-down
```

`hack/test-workloads.yaml` intentionally exercises every code path: a pod-annotated Deployment (ReplicaSet→Deployment chain, pod-source targets), a non-annotated Deployment behind an annotated Service with a named `targetPort` (service-source targets), and a CronJob (Job→CronJob chain).

## Architecture

Two binaries: `cmd/kubescrape` (metadata service) and `cmd/kubescrape-agent` (per-node DaemonSet: log tailer + Prometheus scraper exporting OTLP). Both live in one image; the DaemonSet overrides `command: ["/kubescrape-agent"]`.

### Metadata service

`cmd/kubescrape` serves Kubernetes pod/container metadata over HTTP. Everything is driven by one initial LIST + WATCH per resource (client-go informers); there is **no per-request API-server traffic**.

Data flow:

- **Pod informer** (full objects) → `internal/store.Store`: pods keyed by UID, with two derived indexes — normalized container-ID → entry, and nodeName → pods. `managedFields` are stripped by an informer transform before caching.
- **Service informer** (full objects) → `internal/services.Index`: selector-based matching of Services to pods, for service-annotation scrape discovery.
- **Metadata-only informers** (`PartialObjectMetadata`, cheap) for ReplicaSets, Deployments, Jobs, CronJobs, Namespaces → `internal/owners.Resolver`. Owner chains and namespace metadata are resolved **lazily at request time** from these caches, never stored on the pod record. Adding a new owner kind means extending `owners.AllGVRs` + `kindGVR` and the ClusterRole in `deploy/kubernetes.yaml`.
- **ServiceMonitor CRDs** (opt-in `-servicemonitors`, dynamic informer over unstructured) → `internal/servicemonitors.Index`. A discovery pre-check disables the feature gracefully when the CRD group is absent. Targets resolve monitor → Services (label selector over `services.All(m.ServiceNamespaces())`) → pods (the existing service selector matching), so scraping stays node-local; `scrape.MonitorTargets` handles endpoint `port` (service-port name) / `targetPort` (pod port) / `path` / `scheme` and stamps `Source: "servicemonitor"` + `Monitor: "ns/name"`. Per-endpoint auth/relabelings/intervals are deliberately not interpreted.
- **Events exporter** (opt-in `-events`, `internal/events`): the core Events informer feeds a channel-queued batcher exporting OTLP log records via the shared `otlpexport` client (flags `-otlp-*` on the service). Initial-list history is skipped by timestamp; count bumps re-emit; pod events are enriched from the store (UID cross-checked); delivery is best-effort (queue-full drops count `obs.EventsDropped` with a rate-limited warn, no retries; shutdown flushes the pending batch on a background timeout).
- `internal/server` composes the above per request; `internal/scrape` is pure functions deriving targets from `prometheus.io/*` annotations and ServiceMonitor endpoints; `internal/kubemeta` holds the JSON model and the corev1.Pod → model conversion (`FromPod`). Pods are also resolvable by UID via `GET /v1/pod-uids/{uid}` (`store.GetPodByUID`, tombstone-aware) and by pod IP via `GET /v1/pod-ips/{ip}` (`store.GetPodByIP`, deliberately NOT tombstone-aware — IPs recycle; hostNetwork pods excluded) — the agent's OTLP-ingest enricher uses these.
- **Self-metrics are OTLP-pushed, not scraped**: `internal/obs` defines every internal metric on `metrics.Registry` (`internal/metrics/registry.go` — a prometheus-shaped API over the same `series` storage; no expiry/cardinality caps, `GaugeFunc` evaluated at export; `WithLabelValues` caches typed wrappers per label-value key, so hot-path counter bumps are zero-alloc). Both binaries run `obs.Registry.Run` on `-self-metrics-interval` (default 1m, 0 disables) with their own resource identity (agent: `service.name=kubescrape-agent`+node; service: `service.name=kubescrape`+hostname). `GET /metrics` (both binaries) serves ONLY Go runtime/process metrics (`obs.RuntimeHandler`, a dedicated prometheus registry with the Go+process collectors) — `kubescrape_*` metrics are never exposed there. New failure paths should count into an existing `obs` metric or add one there; tests read values via the `Value()` accessor.
- **HTTP metadata caching**: the service stamps metadata 200s (container/pod/pod-uid/node) with `Cache-Control: max-age` + `ETag` (`server.writeCached`, TTL `-metadata-cache-ttl`, default 10s, 0 disables); `metaclient` caches by URL (stripping the transient `?wait=`), serving fresh entries locally and revalidating stale ones with `If-None-Match` → 304. This cuts repeat lookups on the concurrent ingest and cadvisor paths (`obs.MetadataRequests{cached,not_modified}`). The client cache is mutex-guarded (`metaclient` is shared across concurrent ingest handlers); `Client.now` is injectable for tests.

### Invariants the API guarantees (user requirements — do not regress)

- **Container lookups block per container ID.** An unknown ID registers a waiter channel keyed by that exact ID; the pod-upsert path wakes only those waiters. Multiple concurrent requests for the same ID are all released. Waiting is bounded by `-wait-timeout` (client can shorten via `?wait=`, never lengthen). Requests may legitimately arrive ~1s before the kubelet posts the container ID to the API server — this wait covers that gap.
- **Deleted pods stay resolvable via the container endpoint for `-cache-ttl`** (tombstone: `expireAt` + `deletedAt` stamped, swept periodically). Container IDs replaced by restarts get the same TTL; `lastState.terminated` IDs are indexed while the pod lives.
- **Scrape targets never include deleted pods**: `DeletePod` removes the pod from the node index immediately, before tombstoning. `Succeeded`/`Failed` pods are excluded from targets but remain resolvable by container ID.
- **`prometheus.io/port` is a comma-separated list** (numbers and/or port names) on both pods and Services; service ports translate to pod ports via `targetPort`. Duplicate endpoints reachable via both pod and service annotations are deduped by URL (pod source wins).

### Concurrency model

One `sync.RWMutex` in `store.Store` (plus one in `services.Index`) guards everything; there are no goroutines besides HTTP handlers, the informer callbacks, and the sweeper (`Store.Run`). Stored `kubemeta.Pod` values are never mutated in place — upserts replace whole records — so returning shallow copies under RLock is safe. `FromPod` deep-copies everything it takes from informer objects; never retain or mutate an informer-owned object.

Store tests use an injectable clock (`s.now`); follow that pattern for any time-dependent behavior instead of sleeping.

### Node agent (`internal/agent/...`)

The agent talks only to the metadata service (`metaclient`) and an OTLP gRPC collector (`otlpexport`, built on `go.opentelemetry.io/collector/pdata`) — no Kubernetes API, no RBAC (except `nodes/metrics` for the kubelet scrapes). Resource-attribute building lives in `agent/attrs`: `attrs.Builder` (nil = defaults) runs defaults → static → config templates (funcs: env/coalesce/default/regexMatch/regexReplace; context: Node/Pod/Container/Service) → `attrs.Filter`, driven by the config's `resourceAttributes` section + `-resource-attrs-static`/`-enable`/`-disable`; `attrs.Builders` holds one per pipeline (logs/targets/cadvisor/node/journal/ingest) merged from the section's `pipelines:` subsection. The defaults set `k8s.pod.ip` (from `Pod.PodIP`) as a resource attribute and derive `service.name` via `attrs.ServiceName` (workload owner → pod name), then finish with `attrs.Identity(res)`, which reads whatever identity attrs are on the resource to derive `service.namespace` (= k8s namespace) and `service.instance.id` (fallback: container.id, pod-uid[/container], namespace/pod[/container], node) for Mimir job/instance — neither overwritten if a template set it; it works for pod/container/node AND pre-populated (KSM splitter, ingest, log-metric) resources. **Instance-collision avoidance** (`attrs.PrefixInstance`, cmb-alloy `instance_prefix`): an exporter that DESCRIBES other objects (cadvisor, a KSM splitter) shares the described pod's `service.name`/namespace, so its `service.instance.id` would collide with the pod's own self-scraped metrics (same job+instance, different resource attrs → flapping target_info). The config's `instancePrefix` prepends `prefix-` to the derived instance; it defaults per pipeline via `defaultInstancePrefix` (cadvisor → `"cadvisor"`, others none), precedence **explicit pipeline section > built-in default > top-level base** (so a global base can't strip cadvisor's protection); split rules default it to the describing target's `service.name` (`splitBatcher.defaultPrefix`), overridable per rule (`SplitRule.InstancePrefix`, `""` disables). Split rules also take `dropLabels` (anchored name regex, labels omitted from data points in `putSplitLabels`) and `attributes` (set-if-absent resource fallbacks applied after `Build` in `fillSplitResource`); several groupBy labels may map to one attribute — applied in sorted label order, non-empty overwrites, giving a deterministic coalesce. On KSM/split resources the rule's `datapointAttributes` (config `datapointAttributes`, default `["k8s.node.name"]`, `defaultDatapointAttrs`) are moved off the resource onto the data points (cmb-alloy placement — a described object's node is a property of the object, not the exporter's identity): `splitBatcher.route` removes them from each split resource into `dpAttrs`, and `putSplitLabels` stamps them on every data point (regular self/scrape/cadvisor/node resources keep them as resource attributes). `[]` keeps everything on the resource. It is invoked at every point a final resource is built — tailer metadata resolution, both single-resource batchers, the cadvisor per-pod resources and the splitter resources (which may pre-put identity attrs before Build). New resource-producing code paths must call `Attrs.Build`. `.Node` metadata comes from `GET /v1/nodes/{name}/metadata` via the agent's `startNodeInfo` provider.
- **Series filtering & splitting** (`agent/promscrape/filter.go`, `split.go`, config `metrics` section): ordered first-match-wins keep/drop rules per pipeline applied in `parseAndExport` before conversion (so partial histogram families are possible, as with Prometheus relabeling); splitters route KSM-style targets into per-object resources via `splitBatcher` (groupBy label → attribute mapping + optional metadata enrichment; requires `Kubelet.Meta` even without kubelet scrapes; the shared `podCache` is mutex-guarded because splitters run on concurrent scrape goroutines).
- **Exporter** (`agent/otlpexport`): gRPC or OTLP/HTTP (protobuf POST to /v1/logs, /v1/metrics), bearer token file re-read per minute, TLS CA/skip-verify, bounded retries for metrics only — log retries/rewind belong to the tailer. Payloads are gzip-compressed by default (`Config.Compression`, `-otlp-compression=gzip|none`) via klauspost/compress: `gzip.go` registers a klauspost-backed gRPC "gzip" codec (do NOT import grpc/encoding/gzip too — last init wins) and gzips HTTP bodies with pooled writers into `github.com/JohanLindvall/bufpool` buffers — the pooled buffer IS the request body (the transport's Close recycles it; never set `req.GetBody`, a replay would read recycled memory).
- **Disk buffer** (`agent/spool` + `otlpexport.Buffered`, opt-in `-buffer-dir`/`-buffer-max-bytes`): a Fluent-Bit-style filesystem write-ahead buffer. `spool` is a durable FIFO byte queue — length-prefixed frames in rotating segment files (`<seq>.seg`), a checksummed 24-byte in-place `cursor` (a legacy 16-byte cursor upgrades via rename once), `fsync` per Append, torn-tail truncation on reopen, retire consumed segments; `Pop` surfaces read errors (a vanished head segment is skipped and counted `obs.BufferReadErrors` rather than wedging the drain); `MaxBytes` caps the *undelivered backlog* (`backlog = Σsegsize − readOff`), returning `ErrFull` (tailer rewinds = back-pressure). `Buffered` wraps the client with one spool per signal (generic `sink[T]` over plog/pmetric), implements the shared `otlpexport.Exporter` (ExportLogs+ExportMetrics), enqueues on Export and drains in `Run` with retries (single-attempt inner sends; the backoff persists across cycles to the 30s cap) — commit only after the collector acks (at-least-once across restarts). `IsPermanent` classifies definitive rejections (dropped + counted); auth failures and 404 are deliberately transient — a rotating token or a collector rollout must not drain the backlog into drops. When enabled, main routes ALL consumers (tailer, journald, scrape, log-metrics, ingest) through the single `out` value; the buffer's inner is the raw client. Physical disk can exceed the cap by up to one 8 MiB segment (delivered-but-unreclaimed prefix).
- **Unified config** (`cmd/kubescrape-agent/config.go`, flag `-config`): ALL agent YAML config is one file (`agentConfig`) with optional sections `resourceAttributes` (`attrs.Config`), `logs` (`tailer.SourcesConfig`), `logAttributes` (`logattrs.Config`), `logMetrics` (`metrics.DynamicConfig`), `metrics` (`promscrape.MetricsConfig`). Each section mirrors the standalone file it replaced; the standalone `Load*Config` funcs remain for tests only. Adding a config surface means adding a section here, not a new flag.
- **Log-derived metrics** (`internal/metrics`, config `logMetrics` section, runtime flags `-logs-metrics-interval`/`-max-bytes`/`-name-prefix`): a self-contained package (its former `attr`/`filter` sub-packages are folded in — `labels.go` is the `[]kv` label set with order-independent xxhash folding, `filter.go` the selectors, `series.go` the expiring/cardinality-capped store keyed by `seriesKind`, `dynamic.go` the `Dynamic` spec + label DSL + `DynamicMetricSet`, `export.go` OTLP-only rendering, `linefields.go` the JSON/logfmt line-field fallback). The tailer flush calls `LogMetrics.Add(values, labels, body)`; keys resolve record/resource attrs first (the closures) then fields parsed straight off the line (`keyIndex`/`lineFields`, lightning JSON + logfmt, only the referenced keys). The synthetic `__line__` key (`lineKey`) returns the whole raw line for content selectors/labels. Counter/gauge/histogram/summary (`seriesKind`); gauge has an `action` (set/inc/dec/add/sub + windowed aggregations min/max/avg/first/sum/count/stddev/range/delta, `series.record`+`series.snapshot`+`series.aggregateValue`; aggregations use `sample.sealed` — emit on every export and keep emitting while idle, the first value after an export starts a fresh window; `value`/`value2`/`count` hold the running state — avg=sum/n, count=n, stddev from sum & sum-of-squares, range=max−min, delta=last−first); `valueRegexp` captures the value off the raw line (mutually exclusive with `value`). **Resource grouping**: `Add(values, labels, resource pcommon.Map, line)` takes the line's resource attributes; series are keyed by (resource ⊕ data-point labels ⊕ resourceLabels) — `sample.resource`/`sample.labels`, hashed via `resourceAccum`+`labels.hashAccum`; `Export` groups by `sample.resource` into one ResourceMetrics per distinct resource (`groupByResource`), so the log line's resource attrs are the OTLP resource and the metric's `labels` are data-point attrs. `Dynamic.ResourceLabels` (same DSL) lifts a log-derived label onto the resource; the tailer passes `e.file.resource.Attributes()` (already built by `attrs.Build`, so it carries `service.instance.id`). No stream-labels feature — the resource is the identity. Only the configured metrics are exported (no internal bookkeeping series); rules sharing a name share one series (must agree on type/action). OTLP-only — never reintroduce Prometheus exposition. **Per-line allocation discipline** (bench_test.go; matched-line ≈1 alloc, no-match 0): the pooled `addContext` binds its lookup closures once at construction (per-line state flows through fields, cleared before pool.Put), `matchContext` memoizes via `cached`/`store` (no eval-closure), `keyIndex.want` is precomputed, and JSON line fields decode via `GetPaths` into the reused `lineFields.raws` + `rawScalarString` (raw-token render, no `DecodeAny` boxing; unsafe read-only views into the line string are safe — lightning never mutates its input). The tailer flush mirrors this: one `metricResolver` per flush (bound closures, per-record `rec`/`res` fields), `DynamicMetricSet.Bind` hashes each file's resource once per flush, and `logGrouper.plain` keys attribute-less records by `*file` without `fmt`.
- **Helm chart** (`charts/kubescrape`): every flag is a value; `agent.config` renders verbatim into one mounted ConfigMap (`-config`) with a checksum annotation. Validate with `hack/bin/helm lint` + `helm template`.

- **Tailer performance** (`bench_test.go`): the multiline package (>= v0.0.4) prefilters its start-state regexes with literals derived from the patterns — lines containing no literal skip all start regexes (~12x less per-line CPU). If a pattern change makes the literal set unprovable the matcher silently falls back to full regex evaluation (correct, slower); `TestPrefilterEnabled` (via `patterns.StartLiterals()`) is the alarm. `feedLine` parses the CRI header once and feeds the result to `cri.AddParsed` (multiline >= v0.0.6; the stage's internals never re-parse — the single `cri.Parse` per line is feedLine's), uses precomputed `file.keyStdout`/`keyStderr` (no per-line key concat), and the ledger's offset accounting lives in per-stream `streamState` structs reached via pointers cached on the file (`stStdout`/`stStderr`/`stPlain`, re-derived after every `reset()`) — no map operations per line. BenchmarkIngestLine/BenchmarkIngestFlush track the per-line budget (~0.25µs pipeline, ~1.3µs with enrich).
- **Tailer** (`agent/tailer`): single sweep goroutine over the configured log sources (see the Log sources bullet; default `/var/log/containers`). Log lines flow through the two-stage `github.com/JohanLindvall/multiline` pipeline, per file: `multiline/cri` parses the CRI format and rejoins P/F fragment runs (non-CRI lines pass through rather than being dropped), then `multiline` joins stack traces. Offset mapping relies on stage emission being synchronous inside `Add`/`Flush*`: `lastEnd[key]` is exact because the line completing a run is always the line currently being fed, and `Entry.Lines` (which counts even limit-dropped lines) pops the per-key FIFO of logical-line offset ranges. At-least-once invariants: exports happen inline in the sweep with retries; per-file offsets are committed only after a successful export and never past lines still buffered in either stage (`watermark` = min of `runStart` and FIFO heads); on failure files are rewound to the committed offset and the pipeline is discarded unemitted (the lines get re-read); committed offsets are checkpointed to disk. A file's metadata is resolved (with server-side wait) before its data is consumed — nothing is read until it can be attributed. Rotation: identity = inode + head fingerprint (FNV of first `-logs-fingerprint-bytes`; catches inode reuse AND same-size copytruncate via the modtime-changed-but-no-new-bytes check); rename rotation drains the old fd before switching; deletion drains before drop. **Multi-line groups (CRI P/F runs and stack traces) join across a rename rotation**: if data is still buffered after draining the old inode, `reopen` carries the pipeline instead of flushing it, re-anchors the buffered offsets to the new inode's origin (`reanchor`), and bumps `file.gen` so the already-drained pre-rotation entries (old-inode offsets) don't advance the new inode's checkpoint (flush commits only `e.gen == e.file.gen`). Zero-loss across a crash, including multiple rotations: the rotated-away files are recorded in the checkpoint (`positions.Prefix` list / `file.carried []rotatedPrefix`, one per hop, oldest first; identity = inode+fp + `[from,to)`); on restart (or after a rewind that discarded the buffered group, tracked by `carriedFed`) `feedCarriedPrefix` finds each in the resolved target dir (`findRotated`, by inode+fp) and re-reads them in order into the fresh pipeline before the new inode, reconstructing the group. `carried` clears once the group exports (watermark shows nothing buffered). Caveats: the join only spans rotations the tailer actually **observed** (it follows the symlink, not every intermediate rotated file — a rotation so fast the intermediate inode is never opened loses that segment, pre-existing); and a prefix whose rotated file the runtime already deleted/compressed is skipped (that segment is genuinely gone). Truncation/copytruncate still flush-and-reset (content replaced, joining meaningless). NOTE: this cross-rotation machinery (gen/carried/re-anchor) is the most intricate part of the offset accounting; the cleaner long-term decomposition is a per-container aggregator (logical-stream continuity) separated from a byte-offset durability layer — deferred, not done. Sweeps are fsnotify-driven (watch the symlink dir + resolved target dirs; files directly in the watched dir are a test-only special case) with the poll ticker as fallback. **Log rules** (`Config.Rules` = `metrics.LineFilter`, config `logs.rules`): ordered keep/drop/sample evaluated in flush AFTER enrich (synthetic `__severity__` via `metricResolver.ruleLookup`) and AFTER LogMetrics (metrics see every line); with rules set, records build in a one-record scratch slice and `MoveAndAppendTo` into the batch only when kept; dropped records still advance offsets (an all-dropped batch commits without exporting) and count into `obs.LogRulesDropped`. **Rate limiting** (`Config.RateLimit/RateBurst/RateDrop`, flags `-logs-rate-*`): per-file token bucket in `allowLine`/`consume`; pause mode sets `file.limited` (readFile/readArchive stop reading, pending retried next sweep), drop mode discards lines; rotation drains bypass the limiter (`consume(..., unlimited=true)`) or the rotated remainder would be lost. **Status** (`status.go`): `publishStatus` (housekeeping, `statusEvery` 10s) snapshots per-file positions/lag into an atomic pointer read by `Tailer.Status()` → agent `GET /debug/tailer`; gauges `obs.LogLagBytes{,Total}`. **Pipelined export** (`Config.PipelinedExport`, `-logs-pipelined-export`, opt-in, `pipelined.go`): flush hands the payload+commit-info (`inflight`: offsets/gens/touched) to one worker goroutine and keeps reading; at most one export outstanding; result applied at the next flush, via `pollInflight` in housekeeping (a failure must rewind even when no new lines arrive), or when a rotation/drop `settle(f)`s a touched file FIRST (the rotation machinery must never interleave with an in-flight export); apply is gen-checked, failure rewinds + purges the current batch's entries of rewound files; shutdown settles then goes synchronous.
- **Log sources** (`agent/tailer/sources.go`, config `logs` section): the tailer discovers files by `[]Source` (doublestar include/exclude globs, `**` supported) rather than a single dir; empty config = one default containerd source over `-log-dir`. Each source is `containerd` (CRI filename→containerID→metadata→CRI parse) or plain (arbitrary files: no metadata lookup, resource attrs = builder node attrs + the source's static `attributes`, `service.name` defaulting to the source name; per-source `multiline` override). A file is claimed by the first matching source. Plain files bypass stage-1 (CRI) — `feedPlainLine` feeds the trace stage directly under a single `plainKey` so **the identical rotation / offset / cross-rotation-multiline machinery applies** (`file.source` selects the path; `newPipeline`/`feedLine`/`resolveMetadata`/`stopPipeline` branch on `source.containerd`, guarding nil `criStage`). The watcher watches every source's fixed include base dirs (`scanDirs`). Per-source `compressed` (gzip): compressed files use a separate one-shot `readArchive`/`openArchive` path (offsets in *decompressed* space; gzip isn't seekable so resume re-decompresses and `io.CopyN`-discards the committed prefix; read whole, not skipped-to-end; `rewind`/`drop` branch on `f.compressed`) — the incremental containerd/plain path is untouched when `f.compressed` is false. Caveat: `consume` drops empty physical lines, so multi-line formats needing a blank separator (Go panics) don't join for plain files (CRI's blank is a non-empty `…F ` line); indentation-based traces (Python/Java/.NET) join fine. Non-UTF-8 transcoding was intentionally dropped (Linux-only target) — lines are UTF-8/passthrough. Opt-in `-logs-file-attributes` (`Config.FileAttributes`) stamps `log.file.name` (basename) + `log.file.position` on every record; `entry.start` is the record's start offset (first fragment's, threaded through cri as the entry data / `items[0].start` for trace groups), distinct from `entry.offset` (the post-record commit offset).
- **Scraper** (`agent/promscrape`): hand-written streaming parser for the Prometheus text format, classic + OpenMetrics (constant memory; do NOT replace it with `expfmt`, which buffers whole metric families — the 100k-series requirement is why it exists). The parser classifies samples into roles (`parser.go`); `convert.go` groups histogram/summary component series per family and label set into proper OTLP Histogram/Summary points — grouping state lives only for the current family, so memory stays bounded by the largest family, not the scrape. Exemplars are opt-in (`Config.Exemplars` / `-scrape-exemplars`): enabling them switches the Accept header to OpenMetrics; the parse mode is always detected from the response Content-Type (OM uses float-second timestamps, classic integer milliseconds). Conversion chunks at `BatchPoints` data points per export via the `chunker`/`sink` interfaces. **Hot-path allocation discipline** (bench_test.go guards it; ~4→0 allocs/sample): the parser interns metric/label names and label values per scrape (`internName`/`internValue`, capped — `maxInterned*`; `map[string(b)]` lookups don't allocate) and short-circuits even the intern probes with last-seen memcmp caches (`lastMetric`, per-position `lastKV` — consecutive lines of a family are near-identical; `string(b) == s` doesn't allocate), whitespace is skipped with `skipSpaceTab` (NOT `bytes.TrimLeft`, which builds an ASCII set per call), `converter.labelKey` fingerprints into a reused buffer (`keyBuf`/`keyLbl`) with a keyed `map[string(buf)]` lookup, the batcher's `metricByName`/`remember` cache the last family, and `parseAndExport` filters through `MetricFilter.session()` (per-scrape name→rule-bitmask memo plus a per-(matcher,value) label-regex memo — label values repeat heavily; the shared filter stays immutable). The last-seen caches also skip the byte scans: `parseSample`/`parseLabels` match the previous line's name plus terminator in one memcmp, and `parseQuoted` finds the closing quote with `bytes.IndexByte` — keep new per-sample code free of per-call closures and `fmt`. Per-scrape state is pooled across scrapes: `getParser`/`putParser` reuse the parser + its `bufio.Reader` (family `types` cleared per exposition, intern maps kept warm with cap-based clears), and the converter recycles histogram/summary accumulators via `histFree`/`summFree` — a 10k-series scrape parses in ~3 allocs.
- **Kubelet scrapes** (`agent/promscrape/cadvisor.go`, `cgroup.go`): with `-kubelet-endpoint` the agent scrapes `/metrics/cadvisor` and `/metrics`, bearer-authenticated with the mounted ServiceAccount token (the agent's ServiceAccount exists ONLY for `nodes/metrics`). cadvisor routing (`cadvisorBatcher`) keys resources by the cgroup path in the `id` label (`cgroupIdentity` parses both cgroupfs and systemd layouts): container ID → exact incarnation via `/v1/containers/{id}` (non-blocking); pod-level series (e.g. `container_network_*`, which have NO container id) → `/v1/pods/{ns}/{name}` cross-checked against the cgroup pod UID; sandbox rows (`container="POD"`) fold into the pod resource (their cgroup names the pause container — never look that ID up; `ident.image` is cleared too, it names the pause image). On pod/container-identified rows (`ident.podScoped()`) the redundant `id`/`name`/`image` labels are elided from data points (`redundantOnPodRow` in `putFilteredLabels`; unresolved resources keep the image as `container.image.name`); rollup rows keep `id` — their only distinguisher on the shared node resource. Lookups share a 1-minute TTL cache. `-cadvisor-rollups=false` drops above-pod-level cgroup series AND pod-level rows of container-scoped families; `podScopedFamily` (container_network_*) is the keep-list of families with no per-container breakdown. All pipelines toggle independently: `-logs`, `-metrics`, `-cadvisor`, `-node-metrics`, plus opt-in `-journald`. `Config.HealthMetrics` exports per-target `up`/`scrape_duration_seconds`/`scrape_samples_scraped` gauges after each cycle (`scrapeOutcome` collected in `cycle()`, `exportHealth()`).
- **journald** (`agent/journald`, opt-in `-journald`, **cgo**): reads the journal natively via `github.com/coreos/go-systemd/v22/sdjournal` (libsystemd) — no journalctl subprocess. One resource per unit (identity attrs pre-put, then `Attrs.Build` with the `journal` pipeline). The reader is decoupled behind a `source` interface (`next`/`close`) opened by `Reader.open` (default `openJournal` in `source_sdjournal.go`; tests inject a `fakeSource`), so `journald.go` (batch/convert/cursor/flush) has no cgo and is fully unit-tested. sdjournal seek: resume via `SeekCursor` then skip the (still-present) cursor entry once; tail via `SeekTail`+`Previous`; `next` loops `Next`/`Wait(1s)` honoring ctx. At-least-once via the cursor: committed only after successful export; on failure/reader error the source is closed and reopened from the committed cursor with backoff. Because the agent links libsystemd it is built **CGO_ENABLED=1** (the metadata service stays CGO_ENABLED=0 static); the image is `distroless/base` + copied libsystemd (+ libcap/libgcrypt/libgpg-error/liblz4/liblzma/libzstd), not `distroless/static`.
- **Log enrichment** (`agent/logenrich`, flags `-logs-enrich`/`-journald-enrich`, both default true): `logenrich.Apply(lr, line)` runs `github.com/JohanLindvall/enrich` over each exported line in the tailer and journald `convert`/`flush` paths. Parsed timestamp overrides the CRI/journal time (ingest time stays in ObservedTimestamp), explicit level overrides severity (enrich's severity numbers ARE the OTLP numbers — cast directly), 32/16-hex IDs (dashes stripped, all-zero rejected) fill the trace/span fields, the rest become record attributes (`log.template`, `exception.*`, `azure.*`, `cloud.resource_id`). `exception.stacktrace` is suppressed for `Format == pattern` (a pattern-parsed trace is a verbatim slice of the body — attribute would double megabyte multiline entries; JSON-carried traces stay, and a substring check does NOT work: single-frame JSON traces contain no escapes and are substrings of the raw line). `Result.Format` feeds `obs.LogEnriched{format}` ("" → "none"). Body is never modified. New log-exporting paths should call `Apply` behind a config toggle.
- **Log-line attribute lifting** (`agent/logattrs`, config `logAttributes` section, shared by `-logs` and `-journald`): ordered rules map a JSON/logfmt key in the line to a resource/scope/log attribute. JSON is scanned once for all rule paths via `github.com/JohanLindvall/lightning/pkg/json` `GetPaths`+`DecodeAny` (dotted keys descend into nested objects); logfmt via the `logfmt` reader; do NOT reintroduce `encoding/json`. Resource/scope attributes participate in the flush grouping key (`logGrouper` in the tailer, the `unit\x01res\x01scope` key in journald `convert`) so records with different line-derived resource/scope attrs split into distinct ResourceLogs/ScopeLogs; log-target attrs go on the record. Typed: JSON numbers with no fraction → PutInt, else PutDouble; bool → PutBool.
- **Unified positions** (`agent/positions`, flag `-positions-file`): one JSON file holding both the tailer's per-file `LogPos` map and the journald cursor. The `Store` keeps both sections in memory and rewrites the whole doc atomically under a mutex, so the tailer and journald (different goroutines, independent sections) never clobber each other. It overrides `-checkpoint-file` for logs; the tailer's `checkpoint` type is a `positions.LogPos` alias. **journald has no standalone cursor file** — its cursor persists only through `positions.Store` (`journald.Config.Positions`); without a positions store it starts at the journal tail each run.
- **OTLP ingest** (`agent/otlpingest`, opt-in `-ingest`): a gRPC (`:4317`) + OTLP/HTTP-protobuf (`:4318`, `/v1/logs`+`/v1/metrics`+`/v1/traces`) receiver for telemetry apps push to the local agent. `Enricher` finds a container id (`container.id`/`k8s.container.id`, configurable) or pod uid (`k8s.pod.uid`) on a resource, resolves via `metaclient` (Container = exact incarnation, else `PodByUID`), and merges the `attrs.Builders.Ingest` k8s attributes **without overwriting** anything the sender set. Metrics mode (`-ingest-metrics-mode`): `resource` (id on the resource), `datapoint` (id per point → `split.go` regroups every point into per-object ResourceMetrics, all 5 metric types; points with no id fall back to the resource-level id), or `auto` (resource if every RM has an id, else split). **Traces** (`-ingest-traces`, default true): `EnrichTraces` enriches ResourceSpans the same way; forwarded via `TracesExporter` (`otlpexport.Client.ExportTraces`; `Buffered` passes traces through UNBUFFERED — the pushing sender owns retry). **Peer-IP fallback** (`-ingest-peer-ip-fallback`, opt-in): a resource with NO id resolves via the connection's peer IP (`peer.go`: gRPC `peer.FromContext` / HTTP RemoteAddr → ctx; `Enricher.peerPod` → `metaclient.PodByIP` → `GET /v1/pod-ips/{ip}`; the store's `byPodIP` indexes LIVE non-hostNetwork pods only — deleted pods drop out immediately since pod IPs recycle); counts as `obs.Ingested{peer_ip}`; the splitter's ""-group applies the same fallback. **Batching** (`-ingest-batch-items`/`-ingest-batch-timeout`, `batch.go`): opt-in per-signal coalescing with collector-batch-processor semantics (enqueue acks the sender; bounded queue back-pressures with a retryable error; an oversized payload passes through unsplit; shutdown drains) — pair with `-buffer-dir` for at-least-once. Log bodies additionally run `logenrich.ApplyBody` (non-overwrite variant) when `-ingest-logs-enrich`. Counts into `obs.Ingested{enriched,unresolved,peer_ip}`. Uses the shared `otlpexport` client to forward.
- Tailer tests are timing-based: use `startTailer` (guarantees file creation happens after the initial scan — files present at startup are skipped to their end) and `tl.retryBackoff` to keep retry tests fast. A multi-line group that must survive across sweeps (e.g. straddling a rotation) needs **current** CRI timestamps on its lines (`timeNowCRI()`), not the usual hard-coded `2026-07-05`: the multiline stage's age-out (`FlushBefore`) runs against the real clock, so stale line timestamps flush the group before the continuation arrives.

### E2E in kind

`hack/otel-collector.yaml` deploys a contrib collector (debug exporter). Gotchas learned the hard way: with `verbosity: detailed` the collector's own stdout gets huge and kubelet rotates it within seconds (capture with `kubectl logs -f`, not `--since`); the agent must exclude the observability namespace (`-logs-exclude-namespaces`) or it feeds the collector its own output; after a collector rollout the agent's gRPC connection may keep delivering to the terminating pod for a while.

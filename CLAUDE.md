### End-to-end verification (kind)

Changes to informer wiring, the store, or the HTTP API should be verified against a real cluster, not just unit tests:

```sh
make cluster-up         # 3-node kind cluster; downloads kind/kubectl into hack/bin; deploys hack/test-workloads.yaml
go run ./cmd/kubescrape # uses the kind kubeconfig context
make image && hack/bin/kind load docker-image ghcr.io/johanlindvall/kubescrape:latest --name kubescrape
kubectl apply -f deploy/kubernetes.yaml   # in-cluster deployment (namespace: monitoring)
make cluster-down
```

`hack/test-workloads.yaml` intentionally exercises every code path: a pod-annotated Deployment (ReplicaSetâ†’Deployment chain, pod-source targets), a non-annotated Deployment behind an annotated Service with a named `targetPort` (service-source targets), and a CronJob (Jobâ†’CronJob chain).

## Architecture

Two binaries: `cmd/kubescrape` (metadata service) and `cmd/kubescrape-agent` (per-node DaemonSet: log tailer + Prometheus scraper exporting OTLP). Both live in one image; the DaemonSet overrides `command: ["/kubescrape-agent"]`.

### Metadata service

`cmd/kubescrape` serves Kubernetes pod/container metadata over HTTP. Everything is driven by one initial LIST + WATCH per resource (client-go informers); there is **no per-request API-server traffic**.

Data flow:

- **Pod informer** (full objects) â†’ `internal/store.Store`: pods keyed by UID, with two derived indexes â€” normalized container-ID â†’ entry, and nodeName â†’ pods. `managedFields` are stripped by an informer transform before caching.
- **Service informer** (full objects) â†’ `internal/services.Index`: selector-based matching of Services to pods, for service-annotation scrape discovery.
- **Metadata-only informers** (`PartialObjectMetadata`, cheap) for ReplicaSets, Deployments, Jobs, CronJobs, Namespaces â†’ `internal/owners.Resolver`. Owner chains and namespace metadata are resolved **lazily at request time** from these caches, never stored on the pod record. Adding a new owner kind means extending `owners.AllGVRs` + `kindGVR` and the ClusterRole in `deploy/kubernetes.yaml`.
- **ServiceMonitor CRDs** (opt-in `-servicemonitors`, dynamic informer over unstructured) â†’ `internal/servicemonitors.Index`. A discovery pre-check disables the feature gracefully when the CRD group is absent. Targets resolve monitor â†’ Services (label selector over `services.All(m.ServiceNamespaces())`) â†’ pods (the existing service selector matching), so scraping stays node-local; `scrape.MonitorTargets` handles endpoint `port` (service-port name) / `targetPort` (pod port) / `path` / `scheme` and stamps `Source: "servicemonitor"` + `Monitor: "ns/name"`. Per-endpoint auth/relabelings/intervals are deliberately not interpreted.
- **Events exporter** (opt-in `-events`, `internal/events`): the core Events informer feeds a channel-queued batcher exporting OTLP log records via the shared `otlpexport` client (flags `-otlp-*` on the service). Initial-list history is skipped by timestamp; count bumps re-emit; pod events are enriched from the store (UID cross-checked); delivery is best-effort (queue drops + warns when full, no retries).
- `internal/server` composes the above per request; `internal/scrape` is pure functions deriving targets from `prometheus.io/*` annotations and ServiceMonitor endpoints; `internal/kubemeta` holds the JSON model and the corev1.Pod â†’ model conversion (`FromPod`). Pods are also resolvable by UID via `GET /v1/pod-uids/{uid}` (`store.GetPodByUID`, tombstone-aware) â€” the agent's OTLP-ingest enricher uses this.
- Both binaries serve `GET /metrics` (internal metrics, `internal/obs`, promauto on the default registry â€” the agent under `-listen`, default :8081, with `/healthz`+`/readyz`). New failure paths should count into an existing `obs` metric or add one there.

### Invariants the API guarantees (user requirements â€” do not regress)

- **Container lookups block per container ID.** An unknown ID registers a waiter channel keyed by that exact ID; the pod-upsert path wakes only those waiters. Multiple concurrent requests for the same ID are all released. Waiting is bounded by `-wait-timeout` (client can shorten via `?wait=`, never lengthen). Requests may legitimately arrive ~1s before the kubelet posts the container ID to the API server â€” this wait covers that gap.
- **Deleted pods stay resolvable via the container endpoint for `-cache-ttl`** (tombstone: `expireAt` + `deletedAt` stamped, swept periodically). Container IDs replaced by restarts get the same TTL; `lastState.terminated` IDs are indexed while the pod lives.
- **Scrape targets never include deleted pods**: `DeletePod` removes the pod from the node index immediately, before tombstoning. `Succeeded`/`Failed` pods are excluded from targets but remain resolvable by container ID.
- **`prometheus.io/port` is a comma-separated list** (numbers and/or port names) on both pods and Services; service ports translate to pod ports via `targetPort`. Duplicate endpoints reachable via both pod and service annotations are deduped by URL (pod source wins).

### Concurrency model

One `sync.RWMutex` in `store.Store` (plus one in `services.Index`) guards everything; there are no goroutines besides HTTP handlers, the informer callbacks, and the sweeper (`Store.Run`). Stored `kubemeta.Pod` values are never mutated in place â€” upserts replace whole records â€” so returning shallow copies under RLock is safe. `FromPod` deep-copies everything it takes from informer objects; never retain or mutate an informer-owned object.

Store tests use an injectable clock (`s.now`); follow that pattern for any time-dependent behavior instead of sleeping.

### Node agent (`internal/agent/...`)

The agent talks only to the metadata service (`metaclient`) and an OTLP gRPC collector (`otlpexport`, built on `go.opentelemetry.io/collector/pdata`) â€” no Kubernetes API, no RBAC (except `nodes/metrics` for the kubelet scrapes). Resource-attribute building lives in `agent/attrs`: `attrs.Builder` (nil = defaults) runs defaults â†’ static â†’ config templates (funcs: env/coalesce/default/regexMatch/regexReplace; context: Node/Pod/Container/Service) â†’ `attrs.Filter`, driven by `-resource-attrs-config`/`-static`/`-enable`/`-disable`; `attrs.Builders` holds one per pipeline (logs/targets/cadvisor/node/journal/ingest) merged from the config's `pipelines:` section. It is invoked at every point a final resource is built â€” tailer metadata resolution, both single-resource batchers, the cadvisor per-pod resources and the splitter resources (which may pre-put identity attrs before Build). New resource-producing code paths must call `Attrs.Build`. `.Node` metadata comes from `GET /v1/nodes/{name}/metadata` via the agent's `startNodeInfo` provider.
- **Series filtering & splitting** (`agent/promscrape/filter.go`, `split.go`, `-metrics-config`): ordered first-match-wins keep/drop rules per pipeline applied in `parseAndExport` before conversion (so partial histogram families are possible, as with Prometheus relabeling); splitters route KSM-style targets into per-object resources via `splitBatcher` (groupBy label â†’ attribute mapping + optional metadata enrichment; requires `Kubelet.Meta` even without kubelet scrapes; the shared `podCache` is mutex-guarded because splitters run on concurrent scrape goroutines).
- **Exporter** (`agent/otlpexport`): gRPC or OTLP/HTTP (protobuf POST to /v1/logs, /v1/metrics), bearer token file re-read per minute, TLS CA/skip-verify, bounded retries for metrics only â€” log retries/rewind belong to the tailer.
- **Helm chart** (`charts/kubescrape`): every flag is a value; `agent.attrsConfig`/`agent.metricsConfig` render into mounted ConfigMaps with checksum annotations. Validate with `hack/bin/helm lint` + `helm template`.

- **Tailer** (`agent/tailer`): single sweep goroutine over `/var/log/containers`. Log lines flow through the two-stage `github.com/JohanLindvall/multiline` pipeline, per file: `multiline/cri` parses the CRI format and rejoins P/F fragment runs (non-CRI lines pass through rather than being dropped), then `multiline` joins stack traces. Offset mapping relies on stage emission being synchronous inside `Add`/`Flush*`: `lastEnd[key]` is exact because the line completing a run is always the line currently being fed, and `Entry.Lines` (which counts even limit-dropped lines) pops the per-key FIFO of logical-line offset ranges. At-least-once invariants: exports happen inline in the sweep with retries; per-file offsets are committed only after a successful export and never past lines still buffered in either stage (`watermark` = min of `runStart` and FIFO heads); on failure files are rewound to the committed offset and the pipeline is discarded unemitted (the lines get re-read); committed offsets are checkpointed to disk. A file's metadata is resolved (with server-side wait) before its data is consumed â€” nothing is read until it can be attributed. Rotation: identity = inode + head fingerprint (FNV of first `-logs-fingerprint-bytes`; catches inode reuse AND same-size copytruncate via the modtime-changed-but-no-new-bytes check); rename rotation drains the old fd before switching; deletion drains before drop. Sweeps are fsnotify-driven (watch the symlink dir + resolved target dirs; files directly in the watched dir are a test-only special case) with the poll ticker as fallback.
- **Scraper** (`agent/promscrape`): hand-written streaming parser for the Prometheus text format, classic + OpenMetrics (constant memory; do NOT replace it with `expfmt`, which buffers whole metric families â€” the 100k-series requirement is why it exists). The parser classifies samples into roles (`parser.go`); `convert.go` groups histogram/summary component series per family and label set into proper OTLP Histogram/Summary points â€” grouping state lives only for the current family, so memory stays bounded by the largest family, not the scrape. Exemplars are opt-in (`Config.Exemplars` / `-scrape-exemplars`): enabling them switches the Accept header to OpenMetrics; the parse mode is always detected from the response Content-Type (OM uses float-second timestamps, classic integer milliseconds). Conversion chunks at `BatchPoints` data points per export via the `chunker`/`sink` interfaces.
- **Kubelet scrapes** (`agent/cadvisor.go`, `agent/cgroup.go`): with `-kubelet-endpoint` the agent scrapes `/metrics/cadvisor` and `/metrics`, bearer-authenticated with the mounted ServiceAccount token (the agent's ServiceAccount exists ONLY for `nodes/metrics`). cadvisor routing (`cadvisorBatcher`) keys resources by the cgroup path in the `id` label (`cgroupIdentity` parses both cgroupfs and systemd layouts): container ID â†’ exact incarnation via `/v1/containers/{id}` (non-blocking); pod-level series (e.g. `container_network_*`, which have NO container id) â†’ `/v1/pods/{ns}/{name}` cross-checked against the cgroup pod UID; sandbox rows (`container="POD"`) fold into the pod resource (their cgroup names the pause container â€” never look that ID up). Lookups share a 1-minute TTL cache. `-cadvisor-rollups=false` drops above-pod-level cgroup series AND pod-level rows of container-scoped families; `podScopedFamily` (container_network_*) is the keep-list of families with no per-container breakdown. All pipelines toggle independently: `-logs`, `-metrics`, `-cadvisor`, `-node-metrics`, plus opt-in `-journald`. `Config.HealthMetrics` exports per-target `up`/`scrape_duration_seconds`/`scrape_samples_scraped` gauges after each cycle (`scrapeOutcome` collected in `cycle()`, `exportHealth()`).
- **journald** (`agent/journald`, opt-in `-journald`): runs `journalctl -f -o json` as a subprocess; one resource per unit (identity attrs pre-put, then `Attrs.Build` with the `journal` pipeline). At-least-once via the cursor file: committed only after successful export; on export failure or subprocess exit the process is killed and restarted from the committed cursor with backoff (the tailer-rewind equivalent â€” buffered entries are dropped and re-read). Tests use stub journalctl shell scripts; stubs must `exec sleep` (an orphaned child would hold the test binary's stderr open and stall `go test` 60s). The distroless image has no journalctl â€” document/chart accordingly.
- **Log enrichment** (`agent/logenrich`, flags `-logs-enrich`/`-journald-enrich`, both default true): `logenrich.Apply(lr, line)` runs `github.com/JohanLindvall/enrich` over each exported line in the tailer and journald `convert`/`flush` paths. Parsed timestamp overrides the CRI/journal time (ingest time stays in ObservedTimestamp), explicit level overrides severity (enrich's severity numbers ARE the OTLP numbers â€” cast directly), 32/16-hex IDs (dashes stripped, all-zero rejected) fill the trace/span fields, the rest become record attributes (`log.template`, `exception.*`, `azure.*`, `cloud.resource_id`). `exception.stacktrace` is suppressed for `Format == pattern` (a pattern-parsed trace is a verbatim slice of the body â€” attribute would double megabyte multiline entries; JSON-carried traces stay, and a substring check does NOT work: single-frame JSON traces contain no escapes and are substrings of the raw line). `Result.Format` feeds `obs.LogEnriched{format}` ("" â†’ "none"). Body is never modified. New log-exporting paths should call `Apply` behind a config toggle.
- **Log-line attribute lifting** (`agent/logattrs`, flag `-log-attributes-config`, shared by `-logs` and `-journald`): ordered rules map a JSON/logfmt key in the line to a resource/scope/log attribute. JSON is scanned once for all rule paths via `github.com/JohanLindvall/lightning/pkg/json` `GetPaths`+`DecodeAny` (dotted keys descend into nested objects); logfmt via the `logfmt` reader; do NOT reintroduce `encoding/json`. Resource/scope attributes participate in the flush grouping key (`logGrouper` in the tailer, the `unit\x01res\x01scope` key in journald `convert`) so records with different line-derived resource/scope attrs split into distinct ResourceLogs/ScopeLogs; log-target attrs go on the record. Typed: JSON numbers with no fraction â†’ PutInt, else PutDouble; bool â†’ PutBool.
- **Unified positions** (`agent/positions`, flag `-positions-file`): one JSON file holding both the tailer's per-file `LogPos` map and the journald cursor. The `Store` keeps both sections in memory and rewrites the whole doc atomically under a mutex, so the tailer and journald (different goroutines, independent sections) never clobber each other. When set it overrides `-checkpoint-file` and `-journald-cursor-file`; the tailer's `checkpoint` type is a `positions.LogPos` alias.
- **OTLP ingest** (`agent/otlpingest`, opt-in `-ingest`): a gRPC (`:4317`) + OTLP/HTTP-protobuf (`:4318`, `/v1/logs`+`/v1/metrics`) receiver for telemetry apps push to the local agent. `Enricher` finds a container id (`container.id`/`k8s.container.id`, configurable) or pod uid (`k8s.pod.uid`) on a resource, resolves via `metaclient` (Container = exact incarnation, else `PodByUID`), and merges the `attrs.Builders.Ingest` k8s attributes **without overwriting** anything the sender set. Metrics mode (`-ingest-metrics-mode`): `resource` (id on the resource), `datapoint` (id per point â†’ `split.go` regroups every point into per-object ResourceMetrics, all 5 metric types), or `auto` (resource if every RM has an id, else split). Log bodies additionally run `logenrich.ApplyBody` (non-overwrite variant) when `-ingest-logs-enrich`. Counts into `obs.Ingested{enriched,unresolved}`. Uses the shared `otlpexport` client to forward.
- Tailer tests are timing-based: use `startTailer` (guarantees file creation happens after the initial scan â€” files present at startup are skipped to their end) and `tl.retryBackoff` to keep retry tests fast.

### E2E in kind

`hack/otel-collector.yaml` deploys a contrib collector (debug exporter). Gotchas learned the hard way: with `verbosity: detailed` the collector's own stdout gets huge and kubelet rotates it within seconds (capture with `kubectl logs -f`, not `--since`); the agent must exclude the observability namespace (`-logs-exclude-namespaces`) or it feeds the collector its own output; after a collector rollout the agent's gRPC connection may keep delivering to the terminating pod for a while.
